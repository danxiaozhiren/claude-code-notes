# Claude Code 不是聊天工具，而是 Agentic Harness

## 先把误解说清楚

很多人第一次使用 Claude Code 时，会自然把它放进两个旧框架里：

- 更聪明的代码补全工具
- 能执行命令的聊天机器人

这两个说法都只说中了一小部分。

如果你只问它一段代码是什么意思，它确实像聊天助手。如果你让它补一个函数，它也可能像代码生成工具。但当你给它一个真实工程任务，比如“修复登录失败”“整理这组测试”“把这个 CLI 行为改掉并确保测试通过”，它的行为就会变得完全不同：

```text
读项目
  -> 搜索相关文件
  -> 运行命令或测试
  -> 根据输出修正判断
  -> 编辑代码
  -> 再次验证
  -> 必要时回头重读和重改
```

这不是一次性回答，而是一条持续推进的工作链路。

【官方事实】官方文档把 Claude Code 描述为 agentic coding tool / agentic assistant。它可以读取代码库、编辑文件、运行命令，并和开发工具集成。官方 glossary 还明确区分了 Claude 模型和 Claude Code 这个 agentic harness：模型负责推理，harness 提供工具、上下文管理和执行环境。

【作者推导】所以理解 Claude Code 的第一步，不是问“它能不能回答代码问题”，而是问：

> 它如何把语言模型变成一个可以在工程环境里行动的 agent？

本文只在容易混淆的地方标注来源类型：

- 【官方事实】：来自 Claude Code 官方文档或官方仓库。
- 【作者推导】：基于官方事实整理出的工程判断。
- 【实践练习】：读者可以在自己的环境中复现观察的部分。

## 一、核心认知：Claude Code 的本体是 harness

如果只看模型，Claude 是一个会理解和生成文本的语言模型。

如果只看工具，Claude Code 里有 Read、Edit、Bash、Grep、Glob、WebFetch、LSP 等能力。

但真正让它能完成工程任务的，不是单独某一个模型，也不是单独某一个工具，而是这几部分被组织到同一个循环里：

```text
Claude model
  + tools
  + context management
  + execution environment
  + permissions
  + session state
  = Claude Code agentic harness
```

【官方事实】官方 glossary 对 agentic harness 的定义是：把语言模型变成 coding agent 的工具、上下文管理和执行环境。Claude Code 是 harness，Claude 是里面的模型。

这句话很关键。

这里的 harness 可以先理解成一个承载层：它把模型、工具、权限、context、执行环境绑在一起，让模型的判断能变成可执行、可观察、可回退的工程动作。

它意味着 Claude Code 不是“模型本身变成了 IDE”，而是一个围绕模型建立的运行时。模型负责判断下一步该做什么，harness 负责让这个判断能落地：

| 层 | 负责什么 | 例子 |
| --- | --- | --- |
| Claude model | 理解任务、推理、决定下一步 | 判断应该先看测试还是先看实现 |
| Tools | 行动能力 | 读文件、改文件、跑命令、搜索、查文档 |
| Context management | 工作记忆 | 对话、文件内容、命令输出、CLAUDE.md、memory |
| Execution environment | 运行位置 | 本地终端、IDE、桌面、web、CI |
| Permissions | 行动边界 | 哪些操作需要确认，哪些操作被禁止 |
| Session state | 连续工作 | resume、fork、checkpoints、git state |

【作者推导】如果你把 Claude Code 当成聊天工具，你会期待它“给出正确答案”。如果你把它当成 harness，你会关注它“如何收集证据、采取行动、验证结果、修正路线”。

这两个使用姿势会得到完全不同的体验。

## 二、Agentic loop：它不是一步答完，而是循环推进

【官方事实】官方 How Claude Code works 文档把 Claude Code 的工作方式拆成三个阶段：

- gather context
- take action
- verify results

这三个阶段不是严格线性的流水线，而是会反复交织。

一个简单解释任务可能只需要 gather context：读文件、理解结构、回答问题。

一个 bug 修复任务通常会经历完整循环：

```text
gather context
  读取报错、搜索相关文件、理解调用链

take action
  修改代码、调整测试、更新配置

verify results
  运行测试、检查类型错误、查看命令输出

repeat
  如果失败，带着新输出重新判断下一步
```

【官方事实】官方文档强调，每次工具调用都会把结果反馈给 Claude，并影响下一步判断。比如“修复失败测试”时，Claude 可能先运行测试，再读错误输出，再搜索源文件，再编辑，再重新运行测试。

【作者推导】这不是单纯的产品交互设计，而是语言模型工作方式决定的架构结果。

语言模型在一次生成里，只能基于当前已经进入 context 的内容做推理。它不能在生成答案之前主动“知道”项目里某个文件的最新内容，也不能凭空获得测试输出、git 状态或命令结果。如果这些信息不在 context 里，模型只能猜。

工具调用把信息获取外化成一个可执行步骤：

```text
当前 context
  -> 模型判断还缺什么信息
  -> harness 调用工具获取信息
  -> 工具结果追加进 context
  -> 模型基于新 context 再推理
```

所以 agentic loop 不是“先想完整计划，再照着做”。它更像一种分段推理机制：每一轮只解决当前最值得降低的不确定性，再把真实反馈带回下一轮。

【作者推导】这就是 Claude Code 和“生成一段代码”的根本区别：它不是把答案一次吐出来，而是用工具把不确定性逐步压低。

真正的工程任务几乎都不是静态题。它们会在执行过程中暴露新信息：

- 测试失败的位置和你预想不同
- 项目里已经有类似实现
- 命令输出提示配置缺失
- 类型检查暴露了间接影响
- 代码风格和最初假设不一致

Claude Code 的价值就在于它能把这些新信息接回循环里。

## 三、model 和 harness 的边界

很多误解来自把“Claude 模型会什么”和“Claude Code 能做什么”混在一起。

【官方事实】官方文档说，当文档写“Claude chooses”或“Claude decides”时，指的是模型在做推理。Claude Code 则提供工具、context 管理和执行环境。

可以这样区分：

| 问题 | 更接近 model | 更接近 harness |
| --- | --- | --- |
| 这个 bug 可能在哪里？ | 是 | 否 |
| 应该先读哪个文件？ | 是 | 否 |
| 实际读取文件内容 | 否 | 是 |
| 运行 `npm test` | 否 | 是 |
| 根据测试输出调整方案 | 是 | 否 |
| 保存编辑、管理权限、记录 session | 否 | 是 |

这不是为了做概念游戏，而是为了避免两个错误期待。

第一个错误期待：把 Claude Code 当成“更强模型”。  
如果任务失败，不一定是模型不会，也可能是 context 不够、工具被权限挡住、测试命令不明确、项目没有可验证标准。

第二个错误期待：把 Claude Code 当成“自动脚本”。  
它不是固定流程执行器。它会根据上下文和工具反馈动态选择下一步，所以你给它的目标、约束和验证标准会显著影响行动路径。

【作者推导】更准确的协作方式是：你提供目标和边界，Claude Code 负责在边界内探索路径。

不要这样写：

```text
读 A 文件，然后改 B 文件，然后跑 C 命令。
```

更好的写法是：

```text
登录失败的问题应该和 session 续期有关。
请先理解现有 auth 流程，找到最小修复点。
修完后运行相关测试，不要改无关模块。
```

前者把 Claude Code 当成遥控 shell，后者把它当成会使用工具的工程协作者。

## 四、Claude Code 到底能访问什么

【官方事实】当你在一个目录里运行 `claude`，Claude Code 可以访问当前项目、终端、git state、CLAUDE.md、Auto memory，以及你配置的扩展能力。官方文档还说明，Claude Code 也可以运行在 IDE、桌面、web、Slack、CI/CD 等界面中，但底层 agentic loop 是一致的。

可以把访问范围拆成几类：

| 访问对象 | 含义 | 典型用途 |
| --- | --- | --- |
| 项目文件 | 当前目录及子目录，另有权限时也可访问其他文件 | 读代码、改代码、写文档 |
| 终端 | 你能在命令行运行的构建、测试、git、脚本等 | 跑测试、启动服务、查看状态 |
| git state | 当前分支、未提交改动、最近提交历史 | 理解变更范围、避免覆盖已有改动 |
| CLAUDE.md | 项目级持久指令和约定 | 读取架构说明、命令、代码规范 |
| Auto memory | Claude 自动保存的项目规律和偏好 | 跨 session 恢复部分经验 |
| 扩展能力 | MCP、skills、subagents、浏览器等 | 连接外部系统、加载工作流、隔离调研 |

这里的访问不是“无限访问整台机器”。Claude Code 的默认工作重心是你启动它时所在的项目目录和当前 session 可用的工具环境。超出当前项目的文件、敏感文件、危险命令、外部系统和网络能力，都会受到工具配置、permissions、sandbox、MCP 配置和用户确认流程的影响。

`CLAUDE.md` 可以先理解成项目写给 Claude Code 的持久说明文件；Auto memory 可以先理解成 Claude Code 自动保存的项目偏好和经验笔记。它们如何进入 context，会在第 2 篇专门展开。

这解释了为什么 Claude Code 可以做跨文件工作。

普通补全工具往往只看到当前文件或当前编辑器上下文。Claude Code 则可以主动搜索整个项目，读取多个文件，运行命令，再把结果纳入后续判断。

【作者推导】这也是风险来源。能读、能改、能跑命令，就必须配套权限、检查点和清晰边界。能力越像工程同事，治理方式也越要像工程同事：给任务、给约束、看 diff、跑测试、必要时回滚。

## 五、工具让模型从“会说”变成“会做”

【官方事实】官方 Tools reference 列出了 Claude Code 可用的一组内置工具，包括文件读取、编辑、shell 执行、搜索、LSP code intelligence、subagent、计划模式、MCP resource 读取等。不同工具有不同权限要求。

但工具清单本身不是重点。

重点是：每次工具调用都会产生新的 context。

更准确地说，每次工具调用的输出都会作为新的消息进入当前 context window。所以下一步 Claude 不是凭空继续推理，而是在“看见前一步工具结果”的基础上继续判断。第 3 篇会专门展开工具输出如何动态填充 context，这里先记住这个关系：工具调用带来反馈，反馈进入 context，context 改变下一步决策。

```text
用户：修复 failing tests

Claude 判断：先运行测试
工具输出：某个断言失败

Claude 判断：搜索断言相关代码
工具输出：找到测试和实现文件

Claude 判断：读取实现
工具输出：发现边界条件缺失

Claude 判断：编辑代码
工具输出：diff 已应用

Claude 判断：重新运行测试
工具输出：通过或出现新失败
```

这条链路里，工具不是附属功能，而是推理的一部分。

【作者推导】如果说聊天助手的核心是“回答”，Claude Code 的核心就是“反馈驱动”。它先行动，再从行动结果里获得更多证据，再行动。

所以，给 Claude Code 一个可验证目标很重要。

弱目标：

```text
优化这段代码。
```

强目标：

```text
这个函数在空数组时返回值不对。
请补一个最小测试，再修实现。
最后运行相关测试，保持公开 API 不变。
```

后一种写法给了它验证方向。它知道什么算做完，也知道应该用什么反馈判断自己是否走偏。

## 六、它和三类常见 AI 工具有何不同

【作者推导】Claude Code 最容易被误放进三个旧类别：代码补全、聊天问答、自动化脚本。

但它和这三者的差别都很明显。

| 对比对象 | 典型工作方式 | Claude Code 的不同 | 更适合场景 |
| --- | --- | --- | --- |
| 代码补全 | 在当前位置生成下一段代码 | 可以跨文件搜索、编辑、运行测试 | 写局部实现、补小段代码时补全更轻 |
| 聊天问答 | 根据你提供的上下文回答 | 可以主动获取上下文，而不是只等你粘贴 | 纯概念解释、架构讨论时聊天更直接 |
| 自动化脚本 | 按固定步骤执行 | 会根据工具反馈动态调整路径 | 固定构建、发布、格式化流程用脚本更稳定 |

一个小例子：

```text
任务：修复支付页偶发失败
```

代码补全工具可能等你打开相关文件，然后帮你写局部代码。

聊天助手可能要求你贴报错、贴文件，再给你建议。

自动化脚本只能跑预先写好的诊断步骤。

Claude Code 更像这样：

```text
先搜索 payment 相关文件
读取最近变更
运行测试或复现命令
根据错误输出定位调用链
修改最小范围
重新验证
总结改动和风险
```

适用边界要说清楚：Claude Code 不是永远比前三者更合适。小范围补全、纯概念问答、固定构建流程，仍然可以用更轻的工具。不要为了“用 agent”而把简单任务复杂化。

但当任务需要“理解项目 + 执行动作 + 根据反馈修正”时，Claude Code 的 agentic harness 特征就会显现出来。

## 七、会用 Claude Code 的关键：给它任务，不要只给它指令

【官方事实】官方 How Claude Code works 文档建议把它当作对话来使用：你可以中断它、纠正它、给更具体的约束，也可以让它先探索再实现。文档还建议给 Claude 可验证标准，比如测试用例、预期 UI 或明确输出。

【作者推导】这对应到实际使用，可以总结成四条：

### 1. 给目标

不要只说“看看这段代码”。可以说：

```text
请理解这个模块为什么在 Windows 下路径解析失败。
目标是找到最小修复点，不要重构整个模块。
```

目标决定它要收集什么上下文。

### 2. 给边界

例如：

```text
只改 src/path 目录。
不要修改公开 API。
如果需要新增依赖，先说明理由。
```

边界能减少无关探索，也能降低风险。

### 3. 给验证标准

例如：

```text
修完后运行 pnpm test path-utils。
如果测试命令不存在，请先查 package.json。
```

验证标准会影响它是否停在“看起来改好了”，还是继续确认。

### 4. 允许它先探索

复杂任务不要一上来要求直接改。可以先让它读代码、画出调用链、给计划，再执行。

```text
先不要改文件。
请阅读 auth 相关代码，说明登录态刷新流程，然后给出修复计划。
```

这不是降低效率，而是在信息不足时减少返工。

## 八、安全边界：agentic 不等于无限自治

Claude Code 能做事，不代表应该让它什么都自动做。

【官方事实】官方文档说明 Claude Code 有 checkpoints 和 permissions 两类安全机制。Checkpoints 让文件编辑可回退；permissions 控制哪些操作需要确认，哪些可以放行，哪些应被拒绝。

【官方事实】官方文档还说明，Auto mode 是 research preview。本文不展开 Auto mode，后续讲权限与安全时会单独处理。

【作者推导】对个人开发者来说，最基本的安全姿势是：

- 重要改动前确认 git 工作区状态
- 看 diff，不只看最终总结
- 对高风险命令保持确认
- 把常用测试命令写进 CLAUDE.md
- 对敏感文件、部署命令、生产资源设置明确边界

对团队来说，还要把边界工程化：

- permissions
- hooks
- managed settings
- CI
- code review
- secrets 管理

Claude Code 的能力越强，越不能只依赖“它应该懂”。强约束要交给确定性系统。

## 九、常见坑和修正

### 坑 1：把 Claude Code 当代码补全

错误用法：

```text
帮我写这个函数。
```

更好的用法：

```text
这个函数要兼容空输入和重复 ID。
请先看现有测试风格，补测试后再实现。
```

修正点：从“生成代码”转向“完成可验证任务”。

### 坑 2：把 Claude Code 当聊天机器人

错误用法：

```text
这个项目架构是什么？
```

这当然可以问，但如果你真正想推进工作，更好的问法是：

```text
请阅读 src/auth 和 src/api/session，画出登录态刷新链路。
只输出关键文件、调用关系和潜在风险，先不要改代码。
```

修正点：让它主动收集上下文，而不是只靠你粘贴上下文。

### 坑 3：微操每一步

错误用法：

```text
先 grep A，再 cat B，再改 C，再运行 D。
```

这会把 Claude Code 降级成手动工具代理。即使你知道一条可行路径，也不一定要把每一步都硬编码进提示里。Claude Code 的价值之一，是能在目标和边界内发现更短、更安全或更符合项目习惯的路径。

修正点：指定结果，不要过度指定路径。

### 坑 4：不给验证标准

```text
修一下。
```

这种任务容易停在“表面完成”。

修正：

```text
修完后运行相关单测。
如果没有现成测试，请说明你用了什么方式验证。
```

### 坑 5：忽略权限和上下文成本

Claude Code 能访问项目和终端，也会把工具结果放进 context。大日志、大文件、过多 MCP 工具、模糊任务都可能增加成本和风险。

修正：

- 用 `/context` 观察上下文占用
- 用 permissions 限制敏感操作
- 把长期规则写进 CLAUDE.md
- 把长流程做成 skill

## 十、实践练习

【实践练习】下面的练习不是为了证明 Claude Code “一定会怎样”，而是帮助你观察 agentic harness 如何在自己的环境里工作。不同版本、权限模式、项目结构会带来差异。

每个练习至少保留一种可追溯产物：工具调用摘要、关键命令输出、会话截图、git diff、测试输出或权限确认记录。

### 练习 1：观察一次 bug 修复的 agentic loop

**输入**：在一个小项目里准备一个可复现 bug，让 Claude Code 修复它，并要求保留过程摘要。  
**预期观察**：Claude Code 会经历搜索、读取、编辑、运行测试、根据结果再次调整等多个阶段。  
**验证标准**：保存工具调用摘要、关键命令输出或截图；能把实际行为映射到 gather context、take action、verify results，并指出哪些工具输出改变了下一步决策。

### 练习 2：比较“解释代码”和“修复 bug”的行动链路

**输入**：对同一段代码分别提问“解释一下这段逻辑”和“请修复这里的边界条件 bug，并验证”。  
**预期观察**：解释任务主要集中在读取和总结；修复任务更可能进入编辑、命令执行和验证链路。  
**验证标准**：保存两次任务的工具调用摘要或会话截图，记录工具调用类型、顺序和数量差异。

### 练习 3：给同一任务不同验证标准

**输入**：对同一个小改动分别使用两种提示：一次只说“修复”，一次明确要求“补测试并运行相关测试”。  
**预期观察**：明确验证标准的任务更可能包含测试读取、测试编辑或测试执行步骤。  
**验证标准**：保存两次任务的最终总结、git diff 和测试命令输出，比较验证动作是否不同。

### 练习 4：观察权限如何改变 agent 行为

**输入**：在 Default、Plan mode、Auto-accept edits 等不同模式下执行同一个小任务。  
**预期观察**：确认点、可执行动作、编辑自由度会不同；Plan mode 应更偏向先读和规划。  
**验证标准**：保存每种模式下的权限确认记录、截图或命令输出，说明 Claude Code 是否能编辑、是否能运行命令、是否需要你确认。

> 注意：Auto mode 在官方文档中标注为 research preview。本文不把它作为稳定行为保证；如需观察，请单独记录 Claude Code 版本、权限配置和实验日期。

## 实践复盘建议

做完练习后，不要只看 Claude Code 最后说“完成了”。

建议按三类整理：

- 与官方文档一致：它是否确实围绕 gather context、take action、verify results 循环推进？
- 与预期不同：是权限限制、缺少测试、prompt 太模糊，还是项目结构让它判断失误？
- 文档没有覆盖：保留截图或命令输出，后续版本升级后再复测。

如果你能看懂它的行动链路，就不会只把 Claude Code 当成“能聊天的命令行”。你会开始把它当成一个需要目标、边界和验证标准的工程 agent。
