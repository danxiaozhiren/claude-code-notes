第 1 篇已经建立了最小 loop：

```text
模型请求工具
  -> harness 执行工具
  -> 工具结果回填 messages
  -> 模型继续判断
```

这一篇只问一个问题：

> 工具系统为什么能把模型从“说”变成“做”？

---

## 一、核心认知：工具不是功能清单，而是行动接口

如果没有工具，模型只能输出文本。

比如你让模型“读一下 README 并总结”，它最多回答：

```text
你可以运行 cat README.md，然后把内容发给我。
```

有了工具，模型可以输出一个结构化行动请求：

```text
tool_use: read_file
input: {"path": "README.md"}
```

harness 执行这个请求，把真实文件内容作为 `tool_result` 放回 `messages`。下一轮模型不再凭空猜 README 内容，而是基于真实观察继续推理。

【官方事实】Claude Code 官方 Tools reference 说明，Claude Code 有一组内置工具帮助它理解和修改代码库；工具名也会出现在 permissions、subagent tool lists、hook matchers 等配置里。官方文档还说明，添加自定义工具的方式是连接 MCP server，而 prompt-based workflow 更适合通过 Skill 进入现有 `Skill` 工具。

【社区教学实现】`learn-claude-code` 的 s02 把工具从 s01 的一个 `bash` 扩展到五个：

```text
bash
read_file
write_file
edit_file
glob
```

【作者推导】工具系统的本质不是“多给模型几个按钮”，而是为模型定义一组可执行、可验证、可审计的行动接口。模型负责选择接口，harness 负责实现接口。

这就是“说”和“做”的分界：

| 形态 | 模型输出 | harness 角色 | 结果 |
| --- | --- | --- | --- |
| 普通问答 | 一段建议 | 无 | 用户自己执行 |
| shell 建议 | 一条命令文本 | 无或很弱 | 用户复制执行 |
| tool use | 结构化工具调用 | 执行、校验、回填 | 模型看到真实结果并继续 |

---

## 二、s01 的 bash-only 为什么不够

s01 只有 `bash`。这足够证明 Agent Loop，但不适合复杂工程任务。

因为模型如果想读文件，需要自己拼命令：

```sh
cat path/to/file
```

想写文件，需要自己处理转义、重定向和多行内容：

```sh
cat > hello.py <<'EOF'
print("hello")
EOF
```

想改文件，可能要拼 `sed`、`python -c` 或其他脚本。它不是在表达“我要把这个文件的一段文本换掉”，而是在表达“我猜一个 shell 命令也许能做到这件事”。

【社区教学实现】s02 的 README 直接指出：模型想的是“读这个文件”，却要拼 `cat path/to/file`。这多了一层翻译，浪费 token，也容易拼错。

【作者推导】专用工具的价值不是让模型“更聪明”，而是降低行动表达的阻抗。模型可以直接说出意图：

| 意图 | bash-only 表达 | 专用工具表达 |
| --- | --- | --- |
| 读文件 | `cat README.md` | `read_file(path="README.md")` |
| 写文件 | shell 重定向和转义 | `write_file(path, content)` |
| 局部修改 | `sed` 或脚本 | `edit_file(path, old_text, new_text)` |
| 找文件 | `find` / `ls` / glob shell | `glob(pattern)` |

更明确的接口会带来三个工程收益：

| 收益 | 说明 |
| --- | --- |
| 参数更结构化 | 路径、内容、替换文本分开传，不靠 shell 字符串承载一切 |
| 权限更容易判断 | `write_file` 和 `read_file` 的风险不同，后续可以分别治理 |
| 结果更容易回填 | handler 可以统一截断、格式化、落盘或附加元信息 |

## 三、一个工具分成两半：schema 给模型看，handler 给 runtime 用

工具不是一个函数那么简单。至少要分成两层：

```text
Tool schema
  告诉模型：有什么工具、什么时候用、需要什么参数。

Tool handler
  告诉 harness：收到这个工具请求时，实际怎么执行。
```

【社区教学实现】s02 的工具定义放在 `TOOLS` 数组里。每个工具包含 `name`、`description` 和 `input_schema`。比如 `read_file` 的输入需要 `path`，可选 `limit`。

工具实现则是 Python 函数：

```python
def run_read(path, limit=None):
    ...
```

再用 `TOOL_HANDLERS` 把工具名映射到实现函数：

```python
TOOL_HANDLERS = {
    "bash": run_bash,
    "read_file": run_read,
    "write_file": run_write,
    "edit_file": run_edit,
    "glob": run_glob,
}
```

【作者推导】这是一条很重要的边界：

| 层 | 面向谁 | 关注点 |
| --- | --- | --- |
| schema | 模型 | 这个工具能做什么、参数长什么样 |
| handler | harness | 参数如何校验、动作如何执行、错误如何返回 |
| tool result | 模型下一轮 | 执行后观察到了什么 |

如果 schema 写得模糊，模型会选错工具或传错参数。如果 handler 写得随意，工具会变成安全黑洞。如果 result 太乱，下一轮推理会被噪声污染。

所以工具设计不是“封装函数给 LLM 调”，而是设计一个模型能理解、系统能治理、结果能回填的行动协议。

## 四、dispatch map：为什么新增工具不该改主循环

s01 的工具执行是硬编码的：

```python
output = run_bash(block.input["command"])
```

s02 把它改成查表：

```python
handler = TOOL_HANDLERS.get(block.name)
output = handler(**block.input)
```

这个变化非常小，但意义很大。

【社区教学实现】s02 的 README 把这件事总结成一句话：加一个工具，只需要在 `TOOLS` 里加定义，再在 `TOOL_HANDLERS` 里加映射，循环不用动。

【作者推导】这是 harness 的第一个可扩展性设计。Agent Loop 只负责调度，不关心具体工具怎么实现：

```text
agent_loop
  -> 看到 tool_use
  -> 根据 tool name 找 handler
  -> 执行 handler
  -> 把 output 变成 tool_result
  -> 回填 messages
```

这让系统从“一个 demo”变成“可扩展运行时”。

| 设计 | 后果 |
| --- | --- |
| 每加工具都改 loop | loop 变成 if/else 泥潭，权限、hooks、日志都难统一 |
| 工具通过 dispatch map 注册 | loop 稳定，工具独立演进，外围机制能统一挂载 |

后续 s03 的权限、s04 的 hooks、s19 的 MCP、s20 的完整 harness，都依赖这个分层。因为只有 loop 稳定，工具执行前后才有稳定的挂载点。

## 五、专用工具的真正价值：不是“更方便”，而是“更可治理”

以 s02 的四个文件工具为例。

### 5.1 `safe_path`：工具应该内置边界

【社区教学实现】s02 给文件工具加了 `safe_path()`。它把用户传入路径解析到 `WORKDIR` 内，如果路径逃逸工作区，就返回错误。

这只是教学级保护，不等于完整权限系统。但它说明了一个重要原则：

> 工具 handler 不能假设模型传入的参数天然安全。

【作者推导】工具越接近真实环境，越应该在 handler 层做基础校验：

| 风险 | handler 层能做什么 |
| --- | --- |
| 路径逃逸 | 解析绝对路径后确认仍在 workspace 内 |
| 参数缺失 | 返回结构化错误，而不是让异常泄露 |
| 输出过大 | 截断、摘要、落盘或提示用户 |
| 操作有副作用 | 交给权限层或 hook 层判断 |

### 5.2 `edit_file`：局部编辑比整文件覆盖更可控

s02 的 `edit_file` 用 `old_text` 和 `new_text` 做一次精确替换。

这不是最强大的编辑方式，但它有一个好处：编辑意图非常明确。

```text
在这个文件里，把这一段旧文本换成这一段新文本。
```

【作者推导】相比让模型写 shell 脚本改文件，专用 edit 工具有更好的审查面：

| 维度 | shell 脚本改文件 | `edit_file` 精确替换 |
| --- | --- | --- |
| 意图可读性 | 需要理解脚本 | 旧文本和新文本直接可见 |
| 失败方式 | 可能半执行、误匹配 | 找不到旧文本就返回错误 |
| 权限治理 | 只能看命令字符串 | 可以按路径和工具类型治理 |
| diff 审查 | 间接 | 直接对应文件局部变化 |

这也是工具接口设计的核心：不要只问“它能不能完成任务”，还要问“它失败时是否可理解，执行前是否可判断，执行后是否可验证”。

### 5.3 `glob`：搜索也是行动

很多人会把“行动”理解成写文件、跑命令、发请求。但对 agent 来说，搜索也是行动。

`glob` 不改变文件系统，却改变了模型下一轮能看到的信息。它让模型把“我不知道有哪些文件”变成“我看到了这些候选路径”。

【作者推导】工具系统不只是 action interface，也是 observation interface。一个 coding agent 的常见循环是：

```text
glob 找候选文件
  -> read_file 读取关键文件
  -> edit_file 修改
  -> bash 跑测试
  -> 根据输出继续
```

如果缺少观察类工具，模型会过早猜测。如果缺少改变类工具，模型只能建议。如果缺少验证类工具，模型无法知道自己做对没有。

## 六、多工具调用：一次响应里可以有多个行动请求

在 s02 中，模型可能一次返回多个 `tool_use`。

例如用户说：

```text
读 README.md 和 requirements.txt，然后总结这个项目。
```

模型可能同时请求：

```text
read_file README.md
read_file requirements.txt
```

【社区教学实现】s02 的教学版按 `response.content` 原始顺序逐个执行这些工具调用，把每个结果都收集成 `tool_result`，再一起回填。

```python
for block in response.content:
    if block.type == "tool_use":
        output = handler(**block.input)
        results.append(tool_result)
messages.append({"role": "user", "content": results})
```

【作者推导】多工具调用说明一件事：工具系统不是“每一轮只能做一个动作”的命令行机器人。模型可以在同一轮里提出一组互相独立或顺序相关的观察请求。

但这也带来新的工程问题：

| 问题 | 为什么重要 |
| --- | --- |
| 哪些工具可以并发？ | 两个读文件可以并发，一个写文件和一个读同文件可能不能 |
| 工具结果怎么对应请求？ | 每个 `tool_result` 必须能对应具体 `tool_use_id` |
| 一个失败是否影响其他工具？ | 需要定义错误返回和继续策略 |
| 输出过大怎么办？ | 大输出会占 context，影响后续推理 |

教学版选择顺序执行，是为了概念清晰。生产级 harness 通常需要更细的并发、安全和输出管理，但这些复杂性不改变核心结构：工具请求执行后，结果仍然要回到 `messages`。

## 七、从内置工具到 MCP：外部能力也应该进入同一个工具池

s02 解决的是“手写工具怎么扩展”。到了 s19，问题变成：

> 如果工具不是我写在本进程里的，而是外部服务提供的，怎么办？

比如：

```text
Jira: 查 issue、建 ticket
部署系统: 触发发布、查日志
知识库: 搜文档、创建页面
```

如果每接一个系统都手写工具，harness 会膨胀成一堆专用集成代码。MCP 解决的是标准接入问题。

【官方事实】Claude Code 官方 MCP 文档说明，MCP 用于把 Claude Code 连接到外部工具和数据源。官方 Tools reference 也说明，添加自定义工具可以连接 MCP server。

【社区教学实现】s19 用 `MCPClient` 模拟外部 server 的发现和调用，用 `connect_mcp()` 连接 mock server，再用 `assemble_tool_pool()` 把外部工具组装进工具池。

关键抽象是：

```text
BUILTIN_TOOLS + connected MCP tools
BUILTIN_HANDLERS + mcp handlers
```

工具名用统一前缀避免冲突：

```text
mcp__server__tool
```

例如连接 docs server 后，工具池里会出现类似：

```text
mcp__docs__search
```

【作者推导】这说明 MCP 不是 Agent Loop 之外的另一套系统。站在 harness 视角，它仍然要回到同一个工具池：

| 来源 | 进入模型的形态 | 执行形态 |
| --- | --- | --- |
| 内置工具 | tool schema | 本地 handler |
| MCP 工具 | 带命名空间的 tool schema | MCP client 调外部 server |
| 后台任务工具 | tool schema | handler 启动后台执行并返回占位结果 |

模型不需要知道工具是内置的、mock 的、远程的，还是另一个进程暴露的。它看到的是统一的工具接口；harness 负责路由。

## 八、s20 的视角：工具执行前后会挂上更多控制层

到了 s20，工具系统不再孤立存在。

【社区教学实现】s20 把工具放回完整 harness 的执行链路里：

```text
LLM
  -> has tool_use?
  -> PreToolUse hooks + permission
  -> TOOL_HANDLERS / MCP handlers / background dispatch
  -> PostToolUse hooks
  -> tool_result / task_notification 回 messages
  -> 下一轮
```

这张链路非常关键。

s02 让我们看到：

```text
tool name -> handler -> output
```

s20 让我们看到：

```text
tool use
  -> 执行前拦截
  -> 权限判断
  -> 分发到内置或 MCP handler
  -> 可能进入后台
  -> 执行后 hook
  -> 结果回填
```

【作者推导】这就是为什么第 2 篇要把工具系统讲清楚。后面的权限、hooks、后台任务、MCP、本地/远程执行、结果压缩，本质上都围绕工具调用这条线展开。

如果工具没有统一的入口，权限就只能散落在各个函数里。如果工具没有统一的结果格式，context 管理就无法判断什么应该回填、什么应该压缩、什么应该落盘。如果工具没有统一的命名和 schema，MCP 也无法接进来。

## 九、机制解释：工具系统为什么能让模型“做事”

可以把工具系统压成四步：

```text
1. Schema: 告诉模型有哪些可行动作
2. Selection: 模型根据任务选择工具和参数
3. Handler: harness 校验、执行、捕获结果
4. Feedback: tool_result 回到 messages，影响下一轮
```

这四步缺一不可。

| 缺少什么 | 系统会变成什么 |
| --- | --- |
| 没有 schema | 模型不知道可以请求哪些动作，只能自然语言建议 |
| 没有 handler | 模型能提出动作，但动作不会发生 |
| 没有 result 回填 | 动作发生了，但模型看不到结果 |
| 没有 dispatch 分层 | 每个工具都污染主循环，系统难扩展 |

【作者推导】“模型会用工具”并不是一句魔法。它依赖的是 harness 提供了一套稳定协议：

```text
模型只负责产生结构化行动请求；
harness 负责把请求变成真实世界里的动作；
真实世界的反馈再进入模型上下文。
```

这也是 Agent Harness 和普通 API 编排的差别。普通编排是人写死流程；Agent Harness 是模型在每一轮根据 context 选择下一步工具。工程师要做的不是替模型写完整决策树，而是把行动空间设计得清晰、安全、可观察。

## 十、最小代码理解：抓住 6 个代码点

读 s02、s19、s20 时，先抓住这 6 个点。

| 代码点 | 来自章节 | 看什么 | 为什么重要 |
| --- | --- | --- | --- |
| `TOOLS` | s02 | 工具名、描述、输入 schema | 模型可见的行动接口 |
| `TOOL_HANDLERS` | s02 | 工具名到函数的映射 | loop 不关心具体工具 |
| `safe_path()` | s02 | 路径是否逃逸工作区 | handler 层基础边界 |
| `tool_result` | s02 | 输出如何绑定 `tool_use_id` | 结果能对应回请求 |
| `assemble_tool_pool()` | s19 / s20 | 内置工具和 MCP 工具如何合并 | 外部能力接进同一个工具池 |
| `PreToolUse` / `PostToolUse` | s20 | 工具执行前后挂控制逻辑 | 权限、日志、审计、后处理的入口 |

最小结构图可以写成：

```text
TOOLS / MCP tools
  -> LLM 选择 tool_use
  -> PreToolUse / permission
  -> TOOL_HANDLERS or MCP handlers
  -> PostToolUse
  -> tool_result
  -> messages
  -> 下一轮
```

理解了这张图，后面讲权限、hooks、MCP、context compact 都会自然很多。

## 十一、深水区一：一次工具调用的完整运行轨迹

只看 `TOOLS` 和 `TOOL_HANDLERS`，容易以为工具系统就是“注册几个函数”。真正要理解的是：一次工具调用会穿过模型协议、主循环、handler、结果回填和下一轮推理。

假设用户输入：

```text
Read README.md and summarize what this project is about.
```

在 s02 的结构里，一次典型轨迹可以拆成 9 步：

| 步骤 | 数据形态 | 谁负责 | 关键风险 |
| --- | --- | --- | --- |
| 1. 用户提出目标 | 自然语言 message | 用户 | 目标可能含混，例如没说读哪个目录下的 README |
| 2. harness 带上工具定义调用模型 | `tools=[...]` | `agent_loop` | 工具描述不清会导致模型选错工具 |
| 3. 模型选择工具 | `tool_use(name, input)` | LLM | 参数可能缺失、路径可能不合法 |
| 4. 主循环识别工具块 | content block | `agent_loop` | 如果只处理文本块，行动会丢失 |
| 5. 工具名进入分发层 | `TOOL_HANDLERS[name]` | dispatch map | 未知工具名要能返回可理解错误 |
| 6. handler 执行动作 | `run_read(path)` | runtime | 文件不存在、越权、内容过大 |
| 7. 结果绑定请求 ID | `tool_result(tool_use_id, content)` | harness | 多工具调用时必须对应回原请求 |
| 8. 结果回填上下文 | new message | `agent_loop` | 没有回填，模型看不到真实世界 |
| 9. 下一轮模型总结 | text answer or next tool | LLM | 可能还需要继续读取相关文件 |

用更贴近协议的形态写，就是：

```text
assistant:
  tool_use id=tool_1 name=read_file input={"path":"README.md"}

harness:
  content = run_read(path="README.md")

user:
  tool_result tool_use_id=tool_1 content="<README 的真实内容>"

assistant:
  根据 tool_result 总结项目
```

【作者推导】这里最重要的不是 `read_file` 这个函数，而是“因果链”没有断：

```text
用户目标
  -> 模型选择工具
  -> harness 执行
  -> 结果按 tool_use_id 回填
  -> 模型基于观察继续
```

如果没有 `tool_use_id`，多工具调用时结果就可能对不上。比如模型同时请求读取 `README.md` 和 `pyproject.toml`，两个结果必须能分别回到各自的请求，否则下一轮模型只能看到两段文本，却不知道哪段来自哪里。

这也是为什么工具系统不能简单理解成“模型调用函数”。模型并没有直接执行函数。它只是输出一个结构化意图；真正执行发生在 harness 里。这个边界让工程师可以在执行前做权限检查，在执行后做日志、截断、审计和错误归一化。

## 十二、深水区二：工具失败不是异常结束，而是下一轮观察

很多人写第一个 Agent Harness 时，会把工具失败当成程序异常：

```text
handler 抛异常
  -> agent_loop 退出
  -> 用户看到 traceback
```

这对调试脚本有用，但对 agent 不够好。对 agent 来说，失败本身也可能是下一轮推理需要的观察。

更合适的形态是：

```text
handler 捕获可恢复失败
  -> 返回清晰、短小、可行动的错误文本
  -> tool_result 回填 messages
  -> 模型决定重试、换工具、向用户澄清，或停止
```

不同失败类型应该被分层处理：

| 失败类型 | 例子 | 更好的反馈 | 下一轮模型可以做什么 |
| --- | --- | --- | --- |
| schema 不匹配 | 缺少 `path` | `Error: read_file requires path` | 重新发起带参数的工具调用 |
| 未知工具 | `name=read` 但只注册了 `read_file` | `Error: unknown tool read` | 选择正确工具或停止 |
| 路径越界 | `../../secret.txt` | `Error: path escapes workspace` | 换成工作区内路径 |
| 文件不存在 | `README-ZH.md` 不存在 | `Error: file not found` | 先用 `glob` 搜索候选文件 |
| 替换文本不存在 | `old_text` 没命中 | `Error: text not found` | 先读文件，再生成精确替换 |
| 输出过大 | 读到超长日志 | `Output truncated after N chars` | 缩小范围、搜索关键字、分段读取 |
| 命令超时 | `bash` 长时间不返回 | `Error: command timed out` | 换更窄命令或向用户说明 |
| MCP 未连接 | 调用了不存在的 MCP 工具 | `Error: MCP server not connected` | 先连接 MCP 或退回内置工具 |
| 工具名冲突 | 两个 server 都有 `search` | 使用 `mcp__server__tool` | 明确工具来源 |

【作者推导】一个成熟 harness 的错误处理不只是 `try/except`。它要区分三类错误：

| 类型 | 是否回填给模型 | 处理方式 |
| --- | --- | --- |
| 可恢复的任务错误 | 是 | 变成 `tool_result`，让模型继续判断 |
| 策略拒绝 | 是，但要明确边界 | 例如权限拒绝、路径越界、用户未授权 |
| harness 自身 bug | 不一定 | 记录日志，必要时中断，不能伪装成任务失败 |

这里的边界很关键。`text not found` 是任务观察；`TOOL_HANDLERS` 写错导致 `KeyError` 则是 harness bug。前者应该喂给模型，后者应该修代码。

把失败作为 observation，还有一个好处：它能减少模型“假装完成”。如果 `edit_file` 明确返回 `text not found`，模型下一轮就不应该说“我已经修改好了”，而应该解释没有匹配到文本，或者先读取文件重新定位。

## 十三、深水区三：亲手新增一个 `list_dir` 工具

判断 s02 的结构是否真正可扩展，最直接的办法是新增一个工具，看要不要改主循环。

目标：新增一个只读的 `list_dir` 工具，列出目录下一层条目，不读取文件内容，也不递归。

模型可见的 schema 可以设计成：

```python
{
    "name": "list_dir",
    "description": "List direct children of a directory inside the workspace.",
    "input_schema": {
        "type": "object",
        "properties": {
            "path": {
                "type": "string",
                "description": "Directory path relative to the workspace. Defaults to current directory.",
            }
        },
    },
}
```

runtime handler 可以保持很小：

```python
def run_list_dir(path: str = ".") -> str:
    target = safe_path(path)
    if not target.exists():
        return f"Error: directory not found: {path}"
    if not target.is_dir():
        return f"Error: not a directory: {path}"

    rows = []
    for child in sorted(target.iterdir(), key=lambda p: p.name):
        kind = "dir" if child.is_dir() else "file"
        rows.append(f"{kind}\t{child.name}")
    return "\n".join(rows) if rows else "(empty directory)"
```

然后只需要把它接入映射：

```python
TOOL_HANDLERS = {
    "bash": run_bash,
    "read_file": run_read,
    "write_file": run_write,
    "edit_file": run_edit,
    "glob": run_glob,
    "list_dir": run_list_dir,
}
```

关键观察：`agent_loop` 不需要知道 `list_dir` 是什么。它只需要知道：

```text
拿到 tool_use.name
  -> 找 handler
  -> 传 input
  -> 把 output 包成 tool_result
```

这才叫扩展点。新增工具时，如果你发现自己必须在主循环里加一堆 `if name == "list_dir"`，说明工具抽象还不够干净。

那为什么还需要 `list_dir`，不是已经有 `glob` 了吗？

| 工具 | 更适合的问题 | 不适合的问题 |
| --- | --- | --- |
| `glob` | “找所有 `*.py` 文件” | “快速看当前目录有哪些一级条目” |
| `list_dir` | “看目录轮廓” | “跨目录按模式搜索” |
| `read_file` | “读取已知文件内容” | “探索目录结构” |

【作者推导】工具不是越多越好。`list_dir` 值得加入的前提是：目录盘点是高频动作，而且用 `glob("*")` 表达不够清晰。否则它只是增加模型选择成本。

## 十四、深水区四：工具、Skill、MCP、Hook 怎么分工

工具系统继续扩展后，很容易出现一个问题：什么都想做成工具。

但 Agent Harness 里至少有四类机制，职责不同：

| 机制 | 适合放什么 | 不适合放什么 |
| --- | --- | --- |
| Tool | 稳定、可执行、输入输出清晰的动作 | 长篇流程知识、团队规范 |
| Skill | prompt 型工作流、领域方法、操作规程 | 需要直接访问外部系统的能力 |
| MCP | 外部系统能力、第三方 API、动态工具池 | 本地简单文件操作 |
| Hook / Permission | 确定性控制、审计、拒绝、格式化后处理 | 让模型自己决定是否遵守的规则 |

举几个判断题：

| 需求 | 更适合的机制 | 原因 |
| --- | --- | --- |
| 读取一个文件 | Tool | 动作稳定，输入输出明确 |
| “写一篇符合本仓库风格的文章” | Skill 或仓库文档 | 这是流程和风格知识，不是单一动作 |
| 查询 GitHub issue | MCP | 能力来自外部系统，可能有认证和动态 schema |
| 禁止写出工作区之外的文件 | Permission / handler guard | 这是确定性边界，不能只靠提示词 |
| 每次工具执行后记录日志 | Hook | 与具体工具解耦，适合横切控制 |
| 长输出自动截断并提示模型 | handler 或 PostToolUse | 这是结果治理，不应让模型自由发挥 |

【作者推导】一个有工程弹性的 harness，不会把所有能力都塞进 `TOOLS`。比较稳的分工是：

```text
稳定动作 -> Tool
流程知识 -> Skill
外部能力 -> MCP
确定性边界 -> Permission / Hook
```

这个分工也能解释 s02、s19、s20 的递进关系：

```text
s02: 先把本地工具接口做清楚
s19: 再把外部 MCP 能力接进同一工具池
s20: 最后在工具执行前后挂权限、hooks、上下文和恢复机制
```

所以本文讲的“工具系统”不是终点，而是后面权限、hooks、MCP 和上下文工程的地基。

## 十五、常见坑和修正

### 坑 1：把工具当成 prompt 的补充说明

工具不是“告诉模型可以怎么做”的普通文字说明，而是有 schema、有 handler、有 result 的行动接口。

修正：写工具时同时检查三件事：

```text
模型能不能理解这个工具？
harness 能不能安全执行这个工具？
结果能不能帮助下一轮判断？
```

### 坑 2：工具粒度太粗

如果只给一个 `bash`，模型什么都能做，但什么都难治理。所有风险都藏在一条字符串里。

修正：把高频、可结构化、可审查的动作拆成专用工具，例如读文件、写文件、编辑、搜索、查询文档。

### 坑 3：工具粒度太细

如果每个业务按钮都变成一个工具，模型选择空间会膨胀，schema 也会越来越难理解。

修正：工具应该围绕稳定动作建模，而不是围绕每个临时业务流程建模。流程知识更适合放到 Skill、README、系统 prompt 或任务说明里。

### 坑 4：handler 只追求能跑，不处理失败

工具失败时如果直接抛异常，模型下一轮可能看不到可用信息。

修正：handler 应返回对模型有用的错误，例如：

```text
Error: text not found in README.md
Error: path escapes workspace
Error: command timed out after 120s
```

这些错误本身也是观察结果。

### 坑 5：外部工具接入后不做命名空间

两个外部系统都可能提供 `search`。如果没有命名空间，模型和 harness 都会混乱。

修正：像 s19 那样把外部工具规范化成 `mcp__server__tool` 形态，确保来源清晰。

## 十六、动手实验

### 实验 1：比较 bash-only 和专用工具

**输入**：分别运行 s01 和 s02，对两者都输入：

```text
Read README.md and summarize what this project is about.
```

**预期观察**：s01 需要通过 bash 间接读取文件；s02 更可能直接调用 `read_file`。

**验证标准**：记录两次工具调用形式，说明专用工具如何降低 shell 拼接成本。

### 实验 2：新增一个只读工具

**输入**：在 s02 里新增一个 `list_dir` 工具，只列出目录下文件名，不读文件内容、不写文件。

**预期观察**：你只需要新增 tool schema、handler 函数和 `TOOL_HANDLERS` 映射，`agent_loop` 不需要改。

**验证标准**：保存最小 diff，并说明新增工具时哪些代码属于模型可见 schema，哪些属于 runtime handler。

### 实验 3：观察 handler 错误如何反馈给模型

**输入**：让 s02 编辑一个不存在的文本片段，例如：

```text
In README.md, replace "THIS_TEXT_DOES_NOT_EXIST" with "hello".
```

**预期观察**：`edit_file` 应返回类似 “text not found” 的错误，模型下一轮应基于错误重新判断。

**验证标准**：记录错误结果是否进入最终回答或下一步行动。

### 实验 4：观察 MCP 工具池变化

**输入**：运行 s19，输入：

```text
Connect to the docs MCP server and search for agent loop.
```

**预期观察**：连接 MCP 后，下一轮工具池里应出现带 `mcp__docs__` 前缀的工具。

**验证标准**：记录连接前后可用工具名称变化，说明为什么动态工具池不能只靠启动时固定缓存。

### 实验 5：画出工具执行链路

**输入**：阅读 s20 的 README，把工具执行链路画成自己的图。

**预期观察**：图里应该至少包含 LLM、PreToolUse、permission、handler/MCP handler、PostToolUse、tool_result、messages。

**验证标准**：能解释每个节点解决什么问题，而不是只把名词排成一行。
